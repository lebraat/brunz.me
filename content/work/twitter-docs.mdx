## Context

Twitter's developer platform had grown organically over a decade. By 2019, documentation was fragmented across three separate API product lines — Standard, Premium, and Enterprise — each with its own structure, terminology, and content gaps. Developers couldn't find what they needed, onboarding was confusing, and internal teams were duplicating effort.

I was brought on to fix the information architecture and rebuild the docs from the ground up.

## What I did

**Unified the information architecture** across all three API tiers. This meant designing a single taxonomy that could serve hobbyist developers on the free tier and enterprise clients paying six figures for firehose access — without either group getting lost in content meant for the other.

**Redesigned the developer journey** for each access tier. Rather than organizing docs by API endpoint, I restructured around use cases and progressive disclosure — what you need to know first, what you need to know next, and where to go deep.

**Built tooling and processes** that reduced authorship time by 40% and training time for new contributors by 75%. Documentation became something any product team could maintain, not a bottleneck owned by a single team.

## Impact

- **5.9M developers** served annually across 3 API tiers
- **14% increase** in customer satisfaction scores
- **40% reduction** in documentation authorship time
- **75% reduction** in onboarding time for new doc contributors
- Set an IA model that other teams at Twitter adopted for their own documentation

## Takeaway

Documentation is infrastructure. When the IA is right, developers self-serve, support tickets drop, and every product team ships faster because they're not fighting the docs. This project shaped how I think about developer experience — it's not about writing better words, it's about designing better systems for information.
